# LinkVault Core

LinkVault Core 是一个面向技术研究者、开发者和内容策展人的结构化外链资源聚合与导航系统。该项目并非一个传统的爬虫或采集工具，而是一套围绕特定领域（如编程教程、运维文档、架构设计等）进行高质量外部链接的组织、校验和快速检索的基础设施。其核心价值在于将散落在网络各处的零散技术文章、官方文档和社区讨论，通过标准化的元数据描述和分类体系，转化为可被团队或社区共享的知识索引。

项目的主要目标用户包括技术团队内部的知识库管理员、开源社区文档贡献者以及个人开发者。它解决的核心痛点是技术资料分散、链接失效难以追踪、以及优质内容的重复发现成本高。通过 LinkVault Core，用户可以快速建立一份结构清晰、可维护性强的外部资源清单，并配合自动化脚本检测链接可用性，确保所引用的每一条外部信息都是稳定且可访问的。

## 功能概览

**链接资产管理** 提供统一的链接入库接口，支持批量导入和手工录入，自动解析 URL 构成并提取域名、路径参数等关键信息。

**多维度标签分类** 允许为每条链接赋予技术栈（如 Python、Kubernetes）、内容类型（教程、API 参考、故障排查）和难度等级等自定义标签。

**可用性健康检查** 内置异步 HTTP 探针，可周期性对已收录的 URL 发起 HEAD/GET 请求，自动标记响应异常或超时的失效链接。

**结构化元数据导出** 支持将资源列表导出为 JSON、YAML 或 CSV 格式，便于嵌入到静态站点生成器或 CI/CD 流程中。

**自定义字段扩展** 针对不同场景（如内网文档、公开博客），允许用户新增字段（如作者、发布日期、归档状态）而不修改核心表结构。

**全文检索接口** 基于 SQLite FTS5 或 Elasticsearch 适配器，提供对链接标题、描述和标签的快速关键词查询能力。

**访问日志统计** 记录每条链接在团队内的点击次数与最后访问时间，辅助评估内容热度与价值。

## 应用场景

**技术团队内部知识库的引用管理** 当团队使用 Confluence、Notion 或 MkDocs 搭建内部文档时，LinkVault Core 可作为后端服务统一管理文档中引用的所有外部链接。技术写作者不再需要手动维护一堆散落的 URL，而是通过 REST API 或 CLI 工具引用资源 ID，系统自动渲染可点击的超链接并生成引用脚注。当外部文档更新或迁移时，只需在 LinkVault 中修改一条记录，所有引用该资源的文档页面将同步生效，极大降低维护成本。

**开源项目 README 和官网的外链治理** 成熟的开源项目通常在其 README、贡献指南或官网中附有大量的生态链接（如插件列表、相关项目、学习资料）。这些链接随着时间推移极易出现 404 或域名过期。维护者可以将 LinkVault Core 集成到项目的 CI 流程中，每次合并代码前自动检查所有标记为 `external` 的链接状态，生成健康报告。如果某个链接连续三次检查失败，系统会自动向维护者邮箱发送告警，确保项目文档始终指向可用的资源。

**开发者个人技术雷达的构建** 资深开发者可以利用 LinkVault Core 构建属于自己的技术阅读清单。每天浏览技术社区时，通过浏览器插件将有价值的文章、仓库或视频快速收录到本地实例中。利用标签体系，开发者可以按主题（如云原生、数据库内核）、阅读状态（待读、已读、精读）和评分进行精细化管理。周末时，通过检索接口生成一份本周精选阅读列表，直接渲染为静态 HTML 页面分享给同行。

**在线教育平台的课程外链聚合** 编程课程或技术培训的讲师在准备课件时，需要为学生提供大量补充阅读材料。LinkVault Core 可为每门课程创建独立的资源集合，支持按章节（如"第一章：基础语法"）组织链接。系统生成的对外展示页面是干净的、不带多余导航的纯列表，讲师可直接将页面地址嵌入到课程管理系统（LMS）中。学生访问时，可以看到每条链接的添加备注和推荐优先级，无需反复向讲师询问"哪里有关于 X 的详细资料"。

## 快速开始

以下命令演示如何在本地环境快速启动 LinkVault Core 服务。

```bash
# 克隆项目仓库
git clone https://github.com/your-org/linkvault-core.git
cd linkvault-core

# 安装项目依赖（使用 pipenv 或 poetry）
pip install -r requirements.txt
# 若使用 poetry，执行：poetry install

# 初始化 SQLite 数据库并执行迁移
python manage.py migrate

# 导入示例资源数据（包含本批次收录的链接元数据）
python manage.py loaddata fixtures/sample_links.json

# 启动开发服务器
python manage.py runserver --host 0.0.0.0 --port 8080
```

## 项目结构

```
linkvault-core/
├── src/                                # 核心源代码目录
│   ├── core/                           # 核心业务逻辑模块
│   │   ├── models.py                   # 数据模型定义（Link, Tag, CheckResult）
│   │   ├── validator.py                # URL 规范化与校验器
│   │   └── registry.py                 # 链接注册与生命周期管理
│   ├── checker/                        # 健康检查子模块
│   │   ├── probe.py                    # HTTP 探针实现（异步并发）
│   │   ├── scheduler.py                # 基于 APScheduler 的定时任务
│   │   └── reporter.py                 # 检查报告生成器（HTML/JSON）
│   ├── api/                            # RESTful API 层（FastAPI）
│   │   ├── routes/                     # 路由分组
│   │   │   ├── links.py                # 链接 CRUD 端点
```

## 贡献指南

**提交问题报告** 在 GitHub Issues 页面创建新 Issue 时，请选择对应的模板（Bug 报告或功能请求）。对于 Bug 报告，必须附带可复现的最小步骤、实际结果与预期结果的对比，并贴出相关的日志片段（如有）。对于功能请求，请清晰描述该功能解决的具体场景，避免空泛的"建议增加某某特性"。

**代码贡献流程** 从 `main` 分支创建新的特性分支（命名格式为 `feature/xxx` 或 `fix/xxx`）。确保所有代码通过 `pre-commit` 钩子的检查（包括 black、isort、flake8）。新增或修改功能必须包含对应的单元测试，且测试覆盖率不得低于 85%。提交 Pull Request 前，请将分支变基到最新的 `main` 分支，并解决所有冲突。

**文档贡献** 所有文档位于 `docs/` 目录下，使用 Markdown 编写。新增配置项、API 端点或 CLI 命令时，必须在对应的文档章节同步更新。文档的修改不需要通过严格的测试流程，但需要在 Pull Request 中标注 `[docs]` 前缀，以便维护者快速分类。

**链接资源贡献** 若您希望向公共资源列表（即 `fixtures/sample_links.json`）添加新的技术文章链接，请确保该 URL 指向的内容是稳定可访问的，且与现有资源不重复。提交时，请在 Pull Request 的描述中附上该链接的简要摘要（2-3 句话说明文章核心观点），以便维护者评估其价值。本批次（43/56）的收录已基本完成，新的资源建议针对下一批次提交。

**本地开发环境设置** 贡献者需准备 Python 3.9+ 和 Poetry。克隆仓库后，执行 `poetry install` 安装所有依赖（含开发依赖）。通过 `poetry shell` 激活虚拟环境，运行 `pre-commit install` 安装本地 Git 钩子。所有测试用例可通过 `pytest tests/` 一次性运行。

## 常见问题

**问：LinkVault Core 是否支持 PostgreSQL 或 MySQL 作为生产数据库？**

答：核心数据访问层使用 SQLAlchemy ORM 编写，理论上支持所有 SQLAlchemy 兼容的数据库后端。项目默认使用 SQLite 以降低入门门槛，但在生产环境中推荐切换至 PostgreSQL。切换方法为修改 `config/settings.prod.toml` 中的 `database.url` 配置项，并执行 `python manage.py migrate` 重新迁移。需要注意，PostgreSQL 的全文检索功能比 SQLite FTS5 更强大，因此如果您需要使用高级查询语法，建议在生产环境使用 PostgreSQL。

**问：健康检查探针会不会对目标网站造成负载压力？**

答：探针默认使用 `HEAD` 方法发起请求，仅获取响应头信息而不下载响应体，流量开销极小。并发数通过 `checker.max_workers` 配置项控制，默认值为 5，即同时最多只对 5 个链接发起检查。检查间隔（`checker.interval_hours`）默认设置为 24 小时，避免高频访问。如果目标网站有反爬机制或对 `User-Agent` 敏感，可以在 `checker_rules.yaml` 中自定义请求头策略。一般情况下，该检查行为等同于普通浏览器的一次简单访问，不会对源站构成压力。

**问：我如何将已有的浏览器书签或 Pocket 列表批量导入到系统中？**

答：项目提供了一个独立的导入脚本 `scripts/import_bookmarks.py`，支持解析 Chrome 导出的 HTML 书签文件（`bookmarks.html`）和 Pocket 的 CSV 导出格式。用法示例：`python scripts/import_bookmarks.py --source chrome --file ~/Downloads/bookmarks.html --tag "personal"`。该脚本会读取每个书签的标题、URL 和添加时间，并自动调用注册接口写入数据库。如果您使用的是其他书签服务，建议先导出为通用格式（如 JSON），然后参考 `src/cli/import_cmd.py` 中的实现编写适配器。
