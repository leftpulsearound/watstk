# TechLink Navigator

TechLink Navigator 是一个面向技术研究者和开发者的结构化外链资源聚合平台。该项目专注于收集、分类和呈现来自技术博客、开发社区与工程实践领域的优质深度文章链接，帮助用户在海量信息中快速定位具有阅读价值的技术内容。项目本身不存储任何文章内容，仅提供索引与导航服务，所有版权归属于原始发布者。

本项目定位为技术文档阅读的辅助工具，适用于日常技术调研、架构设计参考、疑难问题排查以及知识体系构建等场景。通过人工筛选与标签化组织，TechLink Navigator 致力于降低技术信息的筛选成本，提升阅读效率。

## 功能概览

**按主题分类导航**：所有外链按照后端开发、前端工程、运维监控、算法与数据结构、安全渗透等一级分类进行组织，支持快速过滤。

**多维度标签系统**：每条链接附带难度等级、阅读时长、技术栈标签，便于根据当前需求匹配阅读内容。

**每日更新队列**：项目维护一个持续更新的待处理链接队列，确保新增资源能够及时进入索引体系。

**全文标题检索**：基于链接的 Article 页面标题信息，提供轻量级的关键词检索能力，支持模糊匹配。

**阅读进度追踪**：为注册用户提供已读/未读标记功能，支持在本地浏览器环境中记录阅读历史。

**导出阅读清单**：支持将当前分类下的链接列表导出为 Markdown 或 JSON 格式，便于离线阅读或分享。

**社区推荐排序**：基于外部引用次数和用户标记热度，对链接进行动态权重排序，优先展示高价值内容。

**RSS 订阅输出**：为每个主分类生成 RSS 订阅源，用户可通过阅读器自动获取新增链接通知。

## 应用场景

**技术选型调研**：当团队需要评估新的技术框架或中间件时，可通过本项目的架构设计分类，快速获取来自一线工程师的实践文章与对比分析，辅助决策。

**故障排查参考**：在线上环境出现异常行为时，运维人员可通过检索关键词找到相似问题的排查记录与解决方案，缩短故障恢复时间。

**技术文章写作素材收集**：技术博主或文档写作者在准备技术分享时，可通过本项目的历史链接库获取多角度的参考资料与案例佐证，丰富文章论据。

**团队新人培训路线图制定**：团队负责人可根据本项目中的入门与进阶分类，为新入职工程师整理一条结构化的技术阅读路径，加速团队融入。

## 快速开始

以下步骤帮助您在本地环境中快速启动 TechLink Navigator 服务。

```bash
# 克隆项目仓库
git clone https://github.com/techlink-navigator/navigator.git
cd navigator

# 安装项目依赖（使用 npm）
npm install

# 启动开发服务器
npm run dev
```

执行上述命令后，服务默认运行于 localhost:3000。浏览器访问该地址即可进入导航首页。若使用 yarn 或 pnpm，请将 npm install 替换为对应的包管理器安装命令。

## 安装要求

| 依赖 | 必需版本 | 说明 |
|------|----------|------|
| Node.js | 18.x 或更高 | 项目运行时环境，建议使用 LTS 版本 |
| npm | 9.x 或更高 | 包管理器，用于安装项目依赖 |
| MongoDB | 5.0 或更高 | 主要数据存储，用于存储链接元数据与用户状态 |

| PM2 | 5.x | 生产环境进程管理，用于守护 Node 服务 |
| Nginx | 1.24 或更高 | 生产环境反向代理与静态资源服务，可选 |

## 贡献指南

我们欢迎并鼓励社区贡献，包括但不限于新增高质量链接、修正分类错误、改进检索算法以及完善文档。请遵循以下流程：

1. 查阅现有链接列表与分类树，确认您希望添加的内容尚未被收录。若存在重叠，优先对已有条目进行补充或更新。

2. 在 GitHub Issues 中提交一个类型为 "link request" 或 "enhancement" 的工单，说明新增链接的原始 URL、建议分类、以及选择该链接的理由（如技术深度、时效性、作者背景等）。

3. 等待核心维护者对该工单进行初审，确认链接内容符合项目收录标准（不包含广告软文、非技术内容、严重过时信息）。初审通过后，您可 fork 仓库并创建新的分支，在 config/seed-data.js 中按格式添加链接条目。

4. 提交 Pull Request 时，请确保在 PR 描述中关联对应的 Issue 编号，并简要说明新增链接的内容概要。CI 流水线将自动运行数据校验与单元测试。

5. 核心维护者将在一周内进行 Code Review，确认无误后合并至主分支。合并后，该链接将出现在下批次更新队列中，并在 24 小时内完成元数据抓取与索引构建。

## 常见问题

**Q: 为什么我点击某些链接后页面显示 404 或内容不存在？**

A: 本项目仅作为外链索引，不托管原始内容。原始博客站点可能会进行内容迁移、删除或调整 URL 格式。若遇到失效链接，请通过 GitHub Issues 提交 "broken link" 报告，维护团队将尽快核实并更新状态。对于已确认失效的链接，我们将从分类导航中移除，但保留在历史批次记录中以供追溯。

**Q: 如何申请将我的技术博客站点加入收录源？**

A: 您可以在 GitHub Issues 中提交类型为 "source request" 的工单，提供您的博客主域名、RSS 订阅地址以及主要技术方向。维护团队将评估内容质量与定位匹配度。由于资源批次有限，并非所有申请都会被接收，但我们会在 15 个工作日内给予明确答复。

**Q: 本地开发时，Elasticsearch 无法启动，是否影响使用？**

A: 不影响核心导航功能。当 Elasticsearch 未就绪时，检索端点会自动降级为基于 MongoDB 全文索引的内存检索模式，但检索速度与结果相关性会有所下降。如果您不需要检索功能，可以在环境配置中将 ENABLE_ES 设为 false 来完全关闭 Elasticsearch 连接尝试。
