# OCCZL Article Index Gateway

OCCZL Article Index Gateway 是一个面向技术文档与博文资源的高性能索引网关，专注于对分散在多个子域名下的历史文章资产进行统一的元数据采集、分类归并与快速检索。本项目并非内容托管平台，而是作为结构化资源导航层，将 wap.blog.occzl.cn 与 www.blog.occzl.cn 两处存量文章条目整合为可编程访问的索引数据集，适用于需要批量回溯、分析或迁移旧有文章系统的开发者与数据维护团队。

项目核心定位为轻量级只读网关，不修改原始源站数据，不缓存文章正文，仅对外暴露稳定的文章元信息映射关系。目标用户包括内部运维人员、第三方内容分析服务以及历史数据归档脚本。通过本项目提供的索引规则与解析示例，用户可在不直接爬取源站的情况下，快速获取文章标识、路径模式与类型标签，从而支撑上层自动化工具链的构建。


## 功能概览

**统一路径映射** 提供基于文章数字标识符的路径规范化输出，将原始 URL 中的大小写混合扩展名（如 .sHtML）统一为逻辑资源键，便于下游系统做幂等处理。

**双域名协同索引** 同时覆盖 wap 与 www 两个子域名下的文章条目，生成跨域名的映射视图，支持按来源子域进行数据分片查询。

**元数据派生标签** 依据文章 ID 的数字特征（长度、前缀、奇偶校验）自动生成轻量级分类标签，如长文标记、历史归档标记、序号段标记，辅助业务侧做粗粒度筛选。

**只读状态检测** 内置源站可达性探测模块，周期性检测每个 URL 的 HTTP 响应状态，并记录响应时间与内容长度，供运维人员监控源站健康度。

**批量导出接口** 提供标准输出格式（JSON Lines / CSV），支持将全部或按条件筛选后的文章索引导出为外部数据文件，便于离线分析或备份。

**路径规范化清洗** 自动去除 URL 中的冗余查询参数（如有），统一使用小写路径格式进行内部存储，同时保留原始 URL 作为原始字段输出，确保可追溯性。


## 应用场景

**历史文章资产盘点** 当团队需要对 blog.occzl.cn 域名下的全部历史文章进行数量统计、时段分布分析或分类整理时，可通过本项目快速获取完整的文章 ID 清单与路径结构，取代人工逐个页面翻查的低效流程。

**自动化监控告警** 运维人员可将本项目作为数据源，定时拉取文章 URL 列表并配合状态检测功能，当发现大量文章返回 4xx 或 5xx 状态码时触发告警，及时发现源站异常或内容被迁移的情况。

**内容迁移辅助工具链** 在进行站点改版或域名合并时，本项目提供的跨子域名索引视图能够清晰展示 wap 与 www 两处的内容重叠与差异情况，帮助迁移脚本准确生成重定向规则或内容去重策略。


## 快速开始

以下步骤指导您在本地环境快速部署并运行本项目的索引服务。

```bash
# 克隆项目仓库
git clone https://github.com/occzl/article-index-gateway.git
cd article-index-gateway

# 安装项目依赖（使用 pip 虚拟环境）
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

# 执行索引构建与本地预览
python main.py --build-index --output-format table
```

执行完毕后，终端将输出当前索引的文章总数、覆盖的域名列表以及前 10 条文章映射记录，并生成一份本地的 index.json 文件供后续使用。

## 项目结构

项目采用模块化分层设计，核心逻辑与辅助工具分离，目录结构如下所示。

```
article-index-gateway/
├── main.py                      # 项目入口脚本，负责解析命令行参数并调度核心流程
├── config/
│   ├── settings.yaml            # 主配置文件，包含源站域名列表、超时阈值、输出路径
│   └── logging.conf             # 日志格式与输出级别配置
├── core/
│   ├── __init__.py
│   ├── indexer.py               # 索引构建核心模块，负责遍历URL并生成映射记录
│   ├── checker.py               # 状态检测模块，封装HTTP请求与响应解析逻辑
│   └── exporter.py              # 导出模块，支持JSON Lines、CSV、表格三种输出格式
├── utils/
│   ├── __init__.py
│   ├── parser.py                # URL解析辅助函数，处理大小写扩展名与路径清洗
│   ├── tagger.py                # 标签生成器，根据文章ID派生分类标签
│   └── logger.py                # 统一日志记录器封装
├── tests/
│   ├── test_indexer.py          # 索引构建模块的单元测试
│   ├── test_checker.py          # 状态检测模块的单元测试
│   └── fixtures/
│       └── sample_urls.json     # 测试用静态URL样本数据
├── docs/
│   ├── getting-started.md       # 入门指南文档
│   ├── configuration.md         # 配置参数详细说明
│   ├── api-reference.md         # 函数接口文档
│   └── operations.md            # 日常运维操作手册
├── output/                      # 运行时生成的索引文件与导出数据存放目录
│   ├── index.json               # 默认生成的完整索引文件
│   └── export/                  # 按用户请求导出的CSV或JSON Lines文件子目录
├── requirements.txt             # Python依赖包清单
├── setup.py                     # 项目安装脚本
└── LICENSE                      # MIT许可证文件
```

第二步：从 main 分支派生一个新的功能分支，分支命名建议采用 feature/描述短语 或 fix/问题编号 的格式。本地开发时请确保所有单元测试通过，并为新增功能补充对应的测试用例。

第三步：提交代码前运行完整的测试套件（pytest tests/）与代码风格检查（flake8 .），确保没有引入回归错误或严重的格式警告。提交信息应使用简洁明确的英文祈使句，说明本次变更的实质内容。

第四步：向 main 分支发起 Pull Request，并在描述中关联相关的 Issue 编号。PR 至少需要一位项目维护者进行 Code Review，确认无误后即可合并。

第五步：合并后的新功能或修复将随下一个版本发布更新至文档与变更日志（CHANGELOG.md）中，贡献者信息会记录在 CONTRIBUTORS.md 文件内。


## 常见问题

**问：索引构建时出现大量连接超时错误，应如何调整？**

答：源站响应速度可能受网络环境或服务器负载影响。您可以在 config/settings.yaml 中调整 request_timeout 参数（默认 10 秒）与 max_retries 参数（默认 3 次）。建议先适当增加超时时间至 30 秒，并将重试次数调整为 5 次。若仍出现大面积超时，请检查网络防火墙策略是否允许对 blog.occzl.cn 域名的出站请求。

**问：导出的索引数据中，部分文章 ID 重复出现，是否表示数据异常？**

答：不会。由于本项目同时索引 wap 与 www 两个子域名，而两个子域名下可能存在文章 ID 相同但内容不同的独立页面（例如 wap 版与桌面版的历史存档）。索引数据中将保留两条记录，并分别标注 source_domain 字段以作区分。下游使用时应根据该字段进行过滤或合并操作。

**问：项目是否支持增量更新而非全量重建索引？**

答：当前稳定版本仅支持全量重建模式，以确保索引数据的一致性与完整性。增量更新功能目前处于实验性分支（feature/incremental-update）中，预计在后续 v2.0 版本中正式合并。如果您有强烈的增量更新需求，可尝试切换至该实验分支并参考 docs/experimental.md 中的使用说明。
