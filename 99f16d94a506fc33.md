# BlogLink Hub

BlogLink Hub 是一个面向技术内容聚合与知识检索的开源外链资源库，专注于对 h5.blog.occzl.cn 平台下大量技术文章与开发笔记进行结构化梳理和统一索引。该项目并非重新托管或复制原始内容，而是建立一套清晰、可扩展的资源导航体系，帮助开发者、技术写作者和研究者在海量碎片化信息中快速定位所需资料。

项目定位为技术资源的中转站与上下文增强层。目标用户包括日常需要查阅技术实现细节的后端工程师、前端开发者、运维人员以及技术内容管理者。通过集中整理文章ID、分类标签与访问路径，本项目显著降低跨文章、跨主题检索时的认知负担，使分散的博客文章形成可关联的知识网络。同时，项目提供标准化的文档模板、自动化索引更新脚本以及本地预览环境，方便社区贡献者参与资源分类与注释工作。

## 功能概览

**统一资源索引**：对指定域名下的文章详情页进行ID级索引登记，每条记录包含唯一数字标识与标准访问路径，确保资源可追溯、可验证。

**分级目录导航**：根据文章内容特征将资源划分为技术栈、算法实践、工程架构、运维记录、语言特性等大类，每个大类下设子标签，支持多维度筛选。

**本地静态检索**：项目内置基于关键词的轻量级检索脚本，可在克隆仓库后离线对已索引文章标题与摘要进行快速匹配。

**变更追踪与差异对比**：通过维护历史版本清单，记录每次新增或移除的资源ID，便于团队审查外部链接的有效性与内容相关性。

**元数据扩展字段**：为每篇文章预留分类、重要程度、阅读时长、关联话题等扩展属性，支持后续接入自动化标注工具或大语言模型辅助打标。

**社区注释系统**：允许贡献者在指定格式的 Markdown 文件中添加对特定文章的阅读笔记、勘误提醒或补充链接，形成集体智慧沉淀。

**一键本地预览**：提供简单的 HTTP 服务脚本，在开发环境启动后可通过浏览器访问索引主页，模拟线上导航站的交互布局。

**导出与集成接口**：支持将索引数据导出为 JSON、CSV 或 YAML 格式，方便嵌入到其他文档站点、监控系统或自动化流程中。

## 应用场景

技术团队内部知识库构建：团队技术负责人可以利用本项目提供的资源清单，快速筛选与团队技术栈匹配的文章，将其纳入内部培训材料或周报推荐阅读列表，减少成员在外部平台无效浏览的时间。

个人开发者学习路径规划：正在系统学习某一技术方向的开发者，可依据资源分类标签从索引中选出该方向下的全部文章，按照基础、进阶、实战的顺序自行安排阅读计划，并结合注释区的经验分享加深理解。

技术博客内容关联分析：技术内容运营人员可借助本项目的分类框架，分析 h5.blog.occzl.cn 平台下文章的主题分布密度与热门话题变迁趋势，为原创选题策划提供数据参考。

离线归档与合规审查：对于需要遵守内部合规要求的企业，项目索引可与爬虫策略配合，定期检查资源链接的可达性及内容变更情况，生成报告用于外部引用审计。

## 快速开始

以下指令适用于 Linux 与 macOS 环境，Windows 用户建议使用 WSL2 或 Git Bash 执行。

```bash
# 克隆项目仓库到本地
git clone https://github.com/your-org/bloglink-hub.git
cd bloglink-hub

# 安装项目依赖（Python 3.8+ 环境）
pip install -r requirements.txt

# 运行本地索引服务（默认监听 8000 端口）
python serve.py --port 8000
```

执行完成后，在浏览器中访问 `http://127.0.0.1:8000` 即可查看当前资源索引主页。如需更新索引数据，可运行 `python update_index.py --source ./data/raw_ids.txt` 从指定清单文件重新生成导航配置。

## 贡献指南

1. 复刻项目仓库至个人账号，在本地创建功能分支（如 `feature/add-resource-123456`），确保分支名称反映本次操作内容。

2. 在 `data/raw_ids.txt` 末尾追加新增的文章ID（每行一个），或修改 `data/categories.yaml` 中已有ID的分类标签与注释字段。若为新增注释，需在 `data/annotations/` 下创建以ID命名的 `.md` 文件，并按模板填写阅读笔记。

3. 执行 `python scripts/update_index.py` 更新索引页面，并通过 `python scripts/check_links.py` 验证所提交的URL是否可访问。若链接失效，应先行确认再提交。

4. 提交变更前运行 `make test` 确保所有单元测试通过，且无格式或解析错误。提交信息应遵循约定式提交规范（如 `feat: add resource 123456` 或 `fix: correct category for 794349`）。

5. 发起拉取请求（Pull Request）到主仓库的 `main` 分支，在请求描述中列出本次新增或修改的资源ID数量及分类调整原因。项目维护者将在两个工作日内完成审核并合并。

## 常见问题

**Q：项目是否存储或缓存文章的实际内容？**

不存储。本项目仅记录资源标识符和访问路径，所有文章内容仍由原始服务器提供。用户访问资源列表中的链接时，将直接跳转至源站阅读。项目不涉及任何内容抓取、转发或二次分发，仅作为导航索引存在。

**Q：如何报告资源链接失效或内容不可访问？**

可在仓库的 Issues 中提交问题，标题注明「链接失效」及对应ID，正文附上访问时间与错误状态码。维护者会定期运行 link check 脚本，对失效链接批量标记并在索引页面添加警示图标。贡献者也可自行修正分类或注释后提交 PR。

**Q：项目是否支持对同一文章ID的多个分类标签？**

支持。在 `data/categories.yaml` 中，每个ID可对应多个标签（以列表形式写入），索引页面将显示所有关联标签。前端筛选逻辑按标签做并集匹配，方便用户从不同维度定位同一篇文章。
