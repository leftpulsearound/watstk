# WebResource Aggregator

WebResource Aggregator 是一个面向技术调研、内容聚合与知识归档场景的轻量级外链资源汇总平台。该项目定位于为开发者、技术写作人员、运维工程师以及数据分析师提供结构化的外部文章索引服务，通过集中管理分散于各技术博客与资讯站点的深度内容，帮助用户快速定位特定主题下的高质量讨论与解决方案。

本项目不对原始内容进行转载或存储，仅提供链接索引与分类导航功能。通过明确的资源分类、稳定的条目ID体系以及简洁的查询接口，用户可以按需检索特定批次、特定来源或特定主题的外链资源，显著降低信息分散带来的时间损耗。当前版本为第32/56批资源整合，共收录180条技术类文章链接。

## 功能概览

**批次化资源管理**：每批次资源均附带批次号与条目总数标识，便于用户追踪内容更新节奏与历史版本。当前批次为第32批，共180条链接。

**原始链接直出**：所有收录的URL均保持用户提供的原始格式输出，不添加协议前缀、不补全域名、不进行大小写转换，确保链接的精确可溯性。

**分类索引视图**：资源列表按来源域名、内容类型或主题标签进行二级分类，用户可按类别浏览或批量导出特定分类下的全部链接。

**轻量化查询接口**：支持按资源ID、关键词匹配或批次号进行检索，查询结果以纯文本或表格形式返回，便于嵌入脚本或进一步处理。

**无冗余元数据**：每条资源仅保留原始URL与序号，不额外存储标题、摘要或标签，降低维护成本并避免信息过载。

**可扩展数据格式**：资源列表采用纯文本行存储格式，每行一条URL，便于使用标准命令行工具（如grep、awk、sed）进行批处理操作。

**跨平台兼容性**：项目核心功能基于Shell脚本与Python标准库实现，可在Linux、macOS及Windows WSL环境中无额外依赖运行。

## 应用场景

技术调研与竞品分析：技术调研人员可定期拉取本项目的资源列表，配合自动化脚本批量打开各文章链接，快速浏览特定技术方向（如后端架构、前端框架、数据库调优）的社区讨论与实践案例，替代逐一搜索的繁琐流程。

知识库外链归档：知识库管理员可将本项目作为外部参考链接的补充源，将批次资源导入内部Wiki或文档系统，为团队提供可追溯的原始讨论上下文，避免重要参考资料散落在个人收藏夹或聊天记录中。

运维监控与链接巡检：运维工程师可利用项目提供的纯文本URL列表，配合自定义健康检查脚本，定期检测各外链的可访问性与响应状态码，及时发现失效链接并更新索引。

数据标注与语料采集：自然语言处理研究人员可将本项目作为语料采集入口，按批次批量抓取文章全文或摘要，构建特定领域的技术文本语料库，用于模型训练或关键词分析。

## 快速开始

```bash
# 克隆项目仓库
git clone https://github.com/your-org/webresource-aggregator.git

# 进入项目目录
cd webresource-aggregator

# 安装依赖（Python 3.6+ 标准库无需额外安装）
# 若使用Shell脚本版本，无需任何依赖

# 运行资源列表导出脚本（生成当前批次全部URL列表）
python3 export_resources.py --batch 32 --output resources_batch_32.txt

# 或使用Shell脚本版本
./export_resources.sh -b 32 -o resources_batch_32.txt

# 按关键词过滤资源（示例：筛选包含特定ID范围的条目）
grep -E "details/[0-9]{4,}" resources_batch_32.txt > filtered_list.txt
```

## 贡献指南

提交新批次资源或更新现有条目时，请遵循以下流程以确保索引质量和可维护性。

第一步：克隆仓库并创建新分支。从主分支检出特性分支，分支命名格式为 `batch-{批次号}-add` 或 `fix-{问题简述}`，避免直接在主干提交。

第二步：准备资源列表文件。在 `data/raw/` 目录下创建新批次文件，命名格式为 `batch_{批次号}_raw.txt`，每行一条URL，确保所有链接均来自允许的域名白名单（参考 `config/source_domains.txt`）。若包含非白名单域名，需提前更新白名单并说明理由。

第三步：运行验证脚本。执行 `scripts/validate_urls.py --file data/raw/batch_{批次号}_raw.txt` 检查链接格式合法性与基本可访问性（HTTP状态码检测）。对于返回4xx或5xx的链接，需确认是否临时失效或永久不可用，并在提交说明中标注。

第四步：更新批次索引。编辑 `config/batch_index.json`，新增当前批次的元数据条目，包含批次号、条目总数、收录日期（UTC）、提交者信息。确保索引文件为合法的JSON格式。

第五步：提交Pull Request。推送分支至远程仓库，创建PR并填写变更摘要模板，包括新增批次号、条目数量、特殊说明（如失效链接、域名变更等）。等待维护者审查合并。

## 常见问题

问：为什么所有资源都来自同一个域名（wap.blog.occzl.cn）？是否会收录其他来源？

答：当前批次为专项资源整合，聚焦于单一技术博客站点的深度文章索引，便于用户集中查阅该站点的全部技术内容。后续批次将逐步引入其他技术社区与资讯平台（如 Medium、Dev.to、Stack Overflow Blog 等），并保持域名白名单的透明更新。用户可在 `config/source_domains.txt` 中查看当前允许的所有来源域名。

问：如果某个链接失效，应该如何处理？

答：项目本身不对外链可用性负责，但提供辅助检测脚本（`scripts/validate_urls.py`）供用户自行运行。若发现失效链接，欢迎提交 Issue 或 Pull Request 标记该条目（在批次文件中添加 `# broken` 注释），或提供可替代的更新链接。维护者会定期审查并清理长期失效条目。

问：如何批量导出所有历史批次的资源列表？

答：使用 `export_resources.py --all --output all_batches.txt` 可导出全部已收录批次的URL，按批次号排序输出。若需按域名过滤，可添加 `--domain` 参数。详细参数请参考 `docs/api-reference.md`。
